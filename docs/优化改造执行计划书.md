# 程序管理器优化改造执行计划书

> 本执行计划书配套 `docs/优化改造设计书.md` 使用，目标是在不引入数据同步能力的前提下，分阶段完成架构治理、统一本地持久化、旧数据升级、性能优化和扩展能力建设。

## 1. 执行原则

1. 先治理基础，再扩展体验。
2. 每个阶段都必须可独立验证。
3. 不破坏已有用户数据。
4. 涉及迁移前必须自动备份。
5. 对外 Tauri 命令优先保持兼容。
6. 数据同步相关能力全部延后，不进入本轮任务。
7. 每轮改造尽量控制改动面，避免一次性大重构。

## 2. 总体阶段

本轮改造分为 6 个阶段：

1. 阶段一：基础准备与版本治理。
2. 阶段二：后端命令拆分与本地存储基础设施。
3. 阶段三：旧数据兼容与升级迁移。
4. 阶段四：前端服务层拆分与统一持久化接入。
5. 阶段五：快捷键、搜索、场景和维护工具优化。
6. 阶段六：本地导入导出、测试补齐和发布验收。

## 3. 阶段一：基础准备与版本治理

### 3.1 目标

建立可控改造基线，统一版本来源，避免后续迁移和发布时出现版本混乱。

### 3.2 任务

1. 盘点当前版本字段：
   - `package.json`
   - `src-tauri/Cargo.toml`
   - `src-tauri/tauri.conf.json`
   - `Config.version`

2. 制定版本规则：
   - 应用发布版本统一为一个来源。
   - 数据结构版本使用独立 `schemaVersion`，不直接等同应用版本。

3. 新增版本检查脚本或文档约束：
   - 发布前检查三处应用版本一致。
   - 数据迁移版本单独记录在 `migrations.json`。

4. 确认当前测试基线：
   - `npm run test`
   - `npm run build`
   - `cd src-tauri && cargo check`

### 3.3 交付物

- 版本规则说明。
- 当前版本差异修复。
- 初始测试基线记录。

### 3.4 验收标准

1. 三处应用版本一致。
2. `Config.version` 和数据 `schemaVersion` 职责清晰。
3. 基线检查命令可执行并记录结果。

## 4. 阶段二：后端命令拆分与本地存储基础设施

### 4.1 目标

拆分后端命令模块，引入统一本地数据仓储，为后续迁移和持久化统一打基础。

### 4.2 任务

1. 拆分 `src-tauri/src/commands/mod.rs`：
   - `commands/config.rs`
   - `commands/apps.rs`
   - `commands/scenes.rs`
   - `commands/maintenance.rs`
   - `commands/integration.rs`
   - `commands/network.rs`

2. 保持命令名称不变：
   - 前端 `invoke` 调用暂不批量改名。
   - `main.rs` 中 `generate_handler` 仍注册相同命令。

3. 新增 `src-tauri/src/storage`：
   - `paths.rs`：数据目录与文件路径。
   - `json_store.rs`：JSON 读写。
   - `atomic_write.rs`：原子写入。
   - `backup.rs`：备份目录和清单。
   - `migration.rs`：迁移记录。

4. 统一数据文件外壳：
   - `schemaVersion`
   - `updatedAt`
   - `data`

5. 引入后端错误类型：
   - 文件不存在。
   - JSON 解析失败。
   - 写入失败。
   - 迁移失败。
   - 权限不足。

### 4.3 交付物

- 拆分后的命令模块。
- `storage` 基础模块。
- 原子写入工具。
- 备份清单结构。

### 4.4 验收标准

1. 现有功能命令可继续调用。
2. `commands/mod.rs` 不再承载具体业务实现。
3. JSON 读写具备单元测试。
4. 原子写入失败不会破坏正式文件。
5. `npm run build` 和 `cargo check` 通过。

## 5. 阶段三：旧数据兼容与升级迁移

### 5.1 目标

兼容现有用户数据，并提供从旧配置和旧 `localStorage` 到新 JSON 文件的升级路径。

### 5.2 任务

1. 兼容 `config.json`：
   - 缺少字段时补默认值。
   - 缺少 `itemType` 时补为 `app`。
   - 缺少设置项时补齐默认设置。
   - 保留已有分类、应用、图标和维护元数据。

2. 迁移旧 base64 图标：
   - 复用现有迁移逻辑。
   - 增加迁移结果记录。
   - 图标迁移失败不阻止应用启动。

3. 增加迁移状态：
   - `migrations.json`
   - 已执行版本。
   - 迁移时间。
   - 迁移结果。
   - 错误摘要。

4. 前端读取旧 `localStorage`：
   - `app_scenes_config`
   - `quick_notes`
   - `todo_schedule`
   - `clipboard_history`

5. 新增迁移命令：
   - `get_legacy_data_status`
   - `migrate_legacy_local_storage`
   - `get_migration_status`

6. 迁移到新文件：
   - `scenes.json`
   - `notes.json`
   - `todos.json`
   - `clipboard.json`
   - `actions.json`

7. 迁移前自动备份：
   - 备份当前 `config.json`。
   - 备份旧 `localStorage` 原始内容。
   - 生成 `manifest.json`。

8. 迁移失败处理：
   - 局部失败记录错误。
   - 未迁移的数据保留在 `localStorage`。
   - 下次启动允许重试。

### 5.3 交付物

- 后端迁移模块。
- 前端旧数据采集逻辑。
- 迁移状态展示或诊断接口。
- 迁移单元测试。

### 5.4 验收标准

1. 旧 `config.json` 可正常加载。
2. 缺失字段会自动补齐。
3. 旧 base64 图标可迁移。
4. 旧场景、便签、待办和剪贴板数据可写入新 JSON。
5. 迁移前存在备份目录。
6. 迁移失败不会删除旧数据。
7. 重启后不会重复迁移已完成数据。

## 6. 阶段四：前端服务层拆分与统一持久化接入

### 6.1 目标

降低 Store 复杂度，将业务规则和外部能力调用移动到 Service 与 Adapter。

### 6.2 任务

1. 新增前端 Adapter：
   - `adapters/tauriAdapter.ts`
   - `adapters/clipboardAdapter.ts`
   - `adapters/shellAdapter.ts`

2. 新增前端 Service：
   - `services/configService.ts`
   - `services/searchService.ts`
   - `services/sceneService.ts`
   - `services/shortcutService.ts`
   - `services/persistenceService.ts`

3. 改造 Store：
   - `appStore.ts` 保留主配置状态和 UI 状态。
   - `scenesStore.ts` 不再直接读写 `localStorage`。
   - `notesStore.ts` 接入新持久化服务。
   - `todoStore.ts` 接入新持久化服务。
   - `clipboardStore.ts` 接入新持久化服务。

4. 配置变更事件：
   - 保存配置后发出 `config-changed`。
   - 搜索窗口、便签窗口、待办窗口按需刷新。

5. 保存队列：
   - 高频写入使用防抖。
   - 窗口关闭或应用退出前强制保存。

### 6.3 交付物

- 前端 Adapter 层。
- 前端 Service 层。
- 接入新 JSON 持久化的 Store。
- 多窗口配置变更事件。

### 6.4 验收标准

1. Store 复杂业务明显减少。
2. 场景、便签、待办、剪贴板不再以 `localStorage` 作为主存储。
3. 多窗口配置变更可以同步。
4. 高频保存不会每次完整立即写入。
5. 旧数据迁移后 UI 展示一致。

## 7. 阶段五：快捷键、搜索、场景和维护工具优化

### 7.1 目标

在基础架构和持久化稳定后，优化核心使用体验和性能。

### 7.2 任务

#### 7.2.1 快捷键可配置化

1. 启动时从配置读取快捷键。
2. 设置变更时重新注册快捷键。
3. 注册失败时恢复旧快捷键。
4. 设置界面增加快捷键录制、冲突提示和恢复默认。

验收标准：

- `Alt+Space`、`Ctrl+K`、`Alt+N`、待办快捷键均可配置。
- 禁用功能后快捷键不会触发窗口。
- 注册失败有明确提示。

#### 7.2.2 快捷搜索优化

1. 建立搜索源接口。
2. 应用搜索建立轻量索引。
3. 搜索窗口打开时加载配置，搜索时不再频繁调用 `load_config`。
4. 新增场景、便签、待办搜索源。
5. 支持命令前缀：
   - `/`
   - `c:`
   - `n:`
   - `t:`
   - `s:`

验收标准：

- 搜索输入过程中不频繁刷新完整配置。
- 应用、剪贴板、场景、便签、待办可搜索。
- 搜索结果执行行为正确。

#### 7.2.3 场景工作流优化

1. 建立动作注册表。
2. 增加动作失败策略。
3. 增加执行状态和执行日志。
4. 支持取消正在执行的场景。
5. 支持复制场景和单个场景导入导出。

验收标准：

- 现有 8 类动作行为不回退。
- 执行失败可选择继续或中断。
- 用户能看到每个动作的执行状态。

#### 7.2.4 维护工具优化

1. 批量检测在锁外执行。
2. 批量任务支持进度事件。
3. 批量删除前展示确认清单。
4. 更新检测支持“接受当前状态为新基准”。
5. 检测结果支持导出日志。

验收标准：

- 大量应用检测时主窗口不明显卡顿。
- 用户能看到准确进度。
- 批量删除前可确认具体条目。

### 7.3 交付物

- 可配置快捷键。
- 搜索源机制。
- 场景动作注册表。
- 维护工具进度和确认机制。

## 8. 阶段六：本地导入导出、测试补齐和发布验收

   - `icons/`

2. 本地导入：

   - 展示导入预览。
   - 自动备份当前数据。
   - 支持覆盖导入和选择性导入。

3. 测试补齐：
   - 配置迁移测试。
   - localStorage 迁移测试。
   - 原子写入测试。
   - 搜索源测试。
   - 场景执行测试。
   - 快捷键配置测试。
   - 维护工具批量任务测试。

4. 发布验收：
   - 执行所有前端测试。
   - 执行前端构建。
   - 执行 Rust 检查。
   - 执行 Tauri 打包。
   - 手动验证旧数据升级。

### 8.3 交付物

- 本地导入导出功能。

### 8.4 验收标准

3. 旧数据升级场景手动验证通过。

推荐顺序：

1. 版本治理。
2. 后端存储基础设施。
3. 配置兼容和迁移。
4. 前端服务层接入新持久化。
5. 快捷键动态注册。
6. 搜索源机制。
7. 场景动作注册表。
8. 维护工具优化。
9. 本地导入导出。
10. 测试和发布验收。

不能提前做的事项：

- 不能在迁移机制完成前移除旧 `localStorage` 读取逻辑。
- 不能在备份机制完成前执行破坏性迁移。
- 不能在命令拆分稳定前大规模改动前端调用名称。
- 不能在快捷键注册失败处理完成前开放用户自定义快捷键。

## 10. 风险与应对

### 10.1 旧配置解析失败

风险：

- 用户本地 `config.json` 被手动修改或半写入。

应对：

- 先备份原文件。
- 尝试宽松解析和字段补齐。
- 失败后使用默认配置启动，并提示用户。

### 10.2 localStorage 迁移失败

风险：

- 旧数据 JSON 损坏或字段结构不符合预期。

应对：

- 按数据类型独立迁移。
- 单类失败不影响其他类别。
- 旧数据不删除，下次可重试。

### 10.3 命令拆分引入行为回退

风险：

- 后端模块拆分后遗漏命令注册或改变错误返回。

应对：

- 对外命令名保持不变。
- 拆分前后执行命令清单对照。
- 先拆文件，再做行为优化。

### 10.4 快捷键注册冲突

风险：

- 用户设置的快捷键被系统或其他应用占用。

应对：

- 注册失败返回明确错误。
- 自动恢复旧快捷键。
- UI 显示冲突提示。

### 10.5 数据文件写入失败

风险：

- 权限不足、磁盘满、杀毒软件拦截。

应对：

- 原子写入。
- 写入失败保留旧文件。
- UI 展示保存失败。
- 日志记录失败原因。

## 11. 回滚策略

每个阶段都必须有回滚点：

1. 阶段一：
   - 版本统一可直接还原版本字段。

2. 阶段二：
   - 命令拆分保持命令名称不变，可按模块回退。

3. 阶段三：
   - 迁移前备份完整数据，可恢复 `config.json` 和旧 `localStorage`。

4. 阶段四：
   - 前端 Store 接入新持久化时保留旧读取兜底。

5. 阶段五：
   - 快捷键注册失败恢复旧快捷键。
   - 搜索源机制可保留原搜索路径作为兜底。

6. 阶段六：
   - 导入前备份当前数据，导入失败恢复备份。

## 12. 验证清单

每个阶段完成后执行：

```bash
npm run test
npm run build
cd src-tauri
cargo check
```

发布前执行：

```bash
npm run tauri build
```

手动验证清单：

1. 旧配置可以启动。
2. 应用分类和图标展示正常。
3. 场景数据从旧 `localStorage` 迁移后正常展示和执行。
4. 便签数据迁移后正常编辑和保存。
5. 待办数据迁移后日期、完成状态和排序正常。
6. 剪贴板历史迁移后搜索和粘贴正常。
7. 快捷键开关和自定义快捷键生效。
8. 快捷搜索能搜索应用、剪贴板、场景、便签和待办。
9. 维护工具批量检测和删除流程正常。
10. 本地导出包可导入恢复。
11. 未出现数据同步相关入口。

## 13. 建议里程碑

完成内容：

- 版本治理。
- 后端命令拆分。
- 本地存储基础设施。
- 旧配置兼容。
- 迁移备份。

完成标志：

- 旧用户数据不会丢。

完成内容：

- 多窗口配置同步事件。

完成标志：

- `localStorage` 不再作为主数据源。
- 多窗口状态同步明显改善。

### 里程碑 3：核心体验优化

完成内容：

- 快捷键可配置。
- 搜索源机制。
- 场景执行状态。
- 维护工具进度和确认机制。

完成标志：

- 高频操作体验提升。
- 自动化场景可观察、可取消。

### 里程碑 4：可发布版本

完成内容：

- 本地导入导出。
- 自动化测试补齐。
- 手动升级验证。

完成标志：
