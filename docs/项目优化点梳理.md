# 项目优化点梳理

> 本文基于当前工作区代码整理，覆盖架构、性能、使用便捷性与后续扩展点，目标是为后续迭代提供可执行的改造路线。

## 1. 项目现状判断

项目是一个面向 Windows 的桌面效率工具，技术栈为 Tauri 2 + Vue 3 + TypeScript + Pinia。当前功能已经覆盖程序启动器、分类管理、场景工作流、剪贴板历史、快捷搜索、快捷便签、待办日程表和维护工具。

当前整体特点：

- 前端主要由 `src/components`、`src/views`、`src/stores` 和 `src/types` 组成，状态管理集中在 Pinia。
- 后端主要由 `src-tauri/src/main.rs`、`src-tauri/src/commands/mod.rs`、`src-tauri/src/utils` 和 `src-tauri/src/models` 组成。
- 主配置存储在 `%APPDATA%/program-manager/config.json`，图标存储在 `%APPDATA%/program-manager/icons`。
- 场景、便签、待办和剪贴板历史仍使用浏览器 `localStorage` 单独持久化。
- 多窗口入口由 `vite.config.ts` 管理，包含主窗口、搜索窗口、便签窗口和待办窗口。

## 2. 评估依据

本次梳理重点阅读了以下文件：

- `package.json`：前端依赖、构建和测试脚本。
- `vite.config.ts`：多窗口构建入口。
- `src-tauri/Cargo.toml`：Rust 依赖和 Windows API 能力。
- `src-tauri/tauri.conf.json`：窗口、托盘、CSP、资产协议和打包配置。
- `src-tauri/src/main.rs`：应用入口、全局快捷键、托盘、多窗口创建。
- `src-tauri/src/commands/mod.rs`：Tauri 命令、程序启动、场景动作、维护功能、更新检测。
- `src-tauri/src/models/mod.rs`：前后端共享配置模型。
- `src-tauri/src/utils/config.rs`：配置读写、图标迁移。
- `src/stores/appStore.ts`：主配置、应用、分类、图标缓存。
- `src/stores/scenesStore.ts`：场景持久化与执行逻辑。
- `src/stores/searchStore.ts`：快捷搜索、计算器、网页搜索、剪贴板搜索。
- `src/stores/clipboardStore.ts`：剪贴板轮询和历史管理。
- `src/stores/notesStore.ts`：快捷便签持久化。
- `src/stores/todoStore.ts`：待办日程表持久化。
- `src/stores/maintenanceStore.ts`：维护面板状态与批量操作。
- `src/types/index.ts` 与相关测试：核心类型、场景动作、快捷动作和条目能力。

## 3. 优先级路线图

### P0：先做的基础治理

1. 拆分后端命令模块，降低 `commands/mod.rs` 的复杂度。
2. 统一持久化策略，减少 `config.json` 与多个 `localStorage` 数据源之间的不一致风险。
3. 收紧脚本执行、CSP 和资产协议范围，降低本地桌面应用的安全风险。
4. 修复版本号管理不一致问题，统一 `package.json`、`Cargo.toml` 和 `tauri.conf.json` 的版本来源。
5. 增加关键路径测试，覆盖配置迁移、场景执行、搜索、快捷键开关和维护工具。

### P1：提升体验和性能

1. 优化快捷搜索配置同步方式，避免每次搜索都主动加载完整配置。
2. 优化剪贴板、便签、待办和场景的数据保存策略，减少频繁完整写入。
3. 将全局快捷键从硬编码改为真正可配置，并处理快捷键冲突。
4. 增加导入、导出、备份和恢复能力。
5. 优化批量验证和更新检测的进度反馈、取消能力和错误展示。

### P2：面向长期扩展

1. 建立插件化或动作扩展机制，降低新增动作类型的改动面。
2. 引入统一搜索源接口，支持应用、剪贴板、便签、待办、场景、网页搜索等来源按需接入。
3. 建立数据迁移框架，支持未来配置结构演进。
4. 抽象跨平台能力层，为后续支持 macOS / Linux 留出边界。
5. 增加使用统计、智能推荐、标签和高级筛选能力。

## 4. 架构可优化点

### 4.1 后端命令模块过于集中

现状：

- `src-tauri/src/commands/mod.rs` 同时承担配置读写、应用管理、程序启动、快捷方式解析、右键菜单、PowerShell 动作、窗口等待、维护检测、背景图片抓取和应用自更新检查。
- 该文件已经成为后端功能汇聚点，后续新增能力会继续放大维护成本。

建议：

- 将命令拆成按业务领域组织的模块：
  - `commands/config.rs`：配置加载、保存、迁移。
  - `commands/apps.rs`：应用增删、启动、管理员启动、路径定位。
  - `commands/scenes.rs`：场景动作需要的系统能力。
  - `commands/maintenance.rs`：有效性检查、更新基准、批量删除。
  - `commands/integration.rs`：右键菜单、快捷方式解析、托盘相关系统集成。
  - `commands/network.rs`：背景图片、自更新等网络能力。
- `commands/mod.rs` 只负责模块导出和命令聚合，避免继续膨胀。
- 对外命令名保持不变，降低前端改动成本。

收益：

- 新增功能时能快速定位模块。
- 后端测试可以按模块补齐。
- 安全审计更清晰，特别是涉及 PowerShell、网络和文件系统的命令。

### 4.2 持久化数据源分散

现状：

- 主配置由 Rust 写入 `config.json`。
- 场景使用 `localStorage` 的 `app_scenes_config`。
- 待办使用 `localStorage` 的 `todo_schedule`。
- 便签使用 `localStorage` 的 `quick_notes`。
- 剪贴板历史使用 `localStorage` 的 `clipboard_history`。

风险：

- 导入导出时难以覆盖完整用户数据。
- WebView 数据清理、重装或迁移时，`localStorage` 数据容易丢失。
- 多窗口之间需要主动刷新或各自读取，容易出现状态不同步。
- 后续做备份、云同步、数据迁移时改造面会变大。

建议：

- 建立统一数据目录，例如：
  - `config.json`：主配置。
  - `scenes.json`：场景配置。
  - `notes.json`：便签。
  - `todos.json`：待办。
  - `clipboard.json`：剪贴板历史。
  - `actions.json`：用户启用的快捷动作。
- 通过 Tauri 命令或轻量数据仓储层统一读写，前端 Store 不直接关心存储位置。
- 对旧 `localStorage` 数据增加一次性迁移逻辑，迁移成功后保留只读兜底一段时间，再清理兼容逻辑。
- 文件写入使用临时文件 + 原子替换，避免进程退出或系统断电导致 JSON 半写入。

收益：

- 用户数据可完整备份。
- 多窗口状态同步更可靠。
- 后续支持导入导出、云同步、数据加密会更自然。

### 4.3 前端 Store 同时承担状态、业务规则和持久化

现状：

- `appStore.ts` 既管理 UI 状态，又直接调用 Tauri 命令，还处理图标 URL 缓存、配置保存、防抖等逻辑。
- `scenesStore.ts` 同时负责场景数据持久化、动作编排、进程判断、PowerShell 脚本拼接和执行。
- `searchStore.ts` 同时负责搜索源编排、计算器、剪贴板、网页搜索和执行动作。

建议：

- 按层拆分：
  - Store：只维护响应式状态和调用用例。
  - Service：承载业务规则，例如场景执行、搜索聚合、配置保存。
  - Adapter：封装 Tauri `invoke`、剪贴板插件、Shell 插件等外部能力。
- 先从风险最高的 `scenesStore.ts` 和 `searchStore.ts` 开始拆分，避免一次性大重构。

收益：

- 单元测试更容易写。
- UI 组件和系统能力解耦。
- 新增窗口、搜索源或动作类型时改动更小。

### 4.4 全局快捷键配置未真正闭环

现状：

- 设置模型中存在 `globalShortcut`、`spotlightShortcut`、`quickNotesShortcut` 等字段。
- `src-tauri/src/main.rs` 中实际快捷键仍硬编码为 `Alt+Space`、`Control+K`、`Alt+N`、`Alt+T`。

建议：

- 启动时从配置读取快捷键并注册。
- 设置变更后，后端注销旧快捷键并注册新快捷键。
- UI 层增加快捷键录制控件、冲突检测和恢复默认按钮。
- 对注册失败给出明确提示，例如快捷键被系统或其他应用占用。

收益：

- 设置项与真实行为一致。
- 用户可以按自己的使用习惯调整快捷键。
- 后续新增窗口或动作快捷键时有统一机制。

### 4.5 版本号来源不一致

现状：

- `package.json` 是 `1.1.0`。
- `src-tauri/Cargo.toml` 是 `1.1.5`。
- `src-tauri/tauri.conf.json` 是 `1.1.5`。

建议：

- 明确单一版本来源。
- 发布脚本中同时更新 `package.json`、`Cargo.toml`、`Cargo.lock` 和 `tauri.conf.json`。
- 应用内更新检查使用同一版本来源，避免前端展示、安装包版本和 Rust 包版本不一致。

收益：

- 降低发布失误概率。
- 更新检测结果更可信。
- 用户和维护者看到的版本一致。

### 4.6 安全边界需要收紧

现状：

- `tauri.conf.json` 中 CSP 允许 `img-src` 和 `connect-src` 使用 `http:`、`https:`。
- `assetProtocol.scope` 是 `["**"]`。
- 多处命令通过 PowerShell 拼接脚本执行，例如打开 URL、打开文件、发送按键、管理员启动、动作模板。

风险：

- 本地资产协议范围过宽，可能扩大本地文件暴露面。
- 字符串拼接脚本需要持续保证转义正确，否则路径、URL 或消息内容可能破坏脚本语义。
- 背景图床和自更新检查属于联网入口，需要更明确的域名、协议和超时策略。

建议：

- 将 `assetProtocol.scope` 收窄到图标目录、背景图目录等明确路径。
- 网络能力按功能拆分白名单，例如背景图片 URL、GitHub Releases 更新检查。
- 对 PowerShell 能力建立统一执行封装：
  - 参数一律通过环境变量或结构化参数传递。
  - 禁止业务层拼接未转义用户输入。
  - 记录命令类型、耗时、退出码和错误。
- 对场景动作中的 URL、路径、进程名和通知文本做统一校验。

收益：

- 降低桌面应用本地文件和脚本执行风险。
- 后续开放自定义动作或插件时有明确安全边界。

## 5. 性能可优化点

### 5.1 快捷搜索每次执行前刷新完整配置

现状：

- `searchStore.performSearch()` 在搜索时调用 `load_config` 刷新完整配置，用于解决独立窗口配置不同步。
- 搜索输入已做 150ms 防抖，但每次实际搜索仍可能触发一次 Tauri 调用和配置反序列化。

建议：

- 主窗口配置变更后通过 Tauri 事件广播给搜索窗口。
- 搜索窗口维护配置快照，只在收到变更事件或窗口打开时刷新。
- 应用列表可构建轻量搜索索引，包含 `id`、名称小写、路径小写、图标 URL。

收益：

- 减少输入过程中的跨进程调用。
- 搜索响应更稳定。
- 应用数量增长后仍能保持低延迟。

### 5.2 剪贴板历史使用轮询和线性搜索

现状：

- 剪贴板历史每 1200ms 轮询一次。
- 去重和搜索都基于数组线性遍历。
- 每次保存会将完整历史写入 `localStorage`。

建议：

- 在 Windows 侧优先评估系统剪贴板监听事件，减少持续轮询。
- 保留轮询作为兜底时，支持用户配置轮询间隔或仅在功能窗口开启时加强频率。
- 使用内容哈希进行快速去重，避免长文本直接全量比较。
- 剪贴板搜索构建小写缓存字段或简单倒排索引。
- 对历史保存继续保留防抖，但迁移到文件持久化后应支持截断、压缩或按条目增量保存。

收益：

- 降低后台资源占用。
- 剪贴板历史变多后搜索不会明显退化。
- 长文本场景下去重成本更低。

### 5.3 图标 URL 预加载可能带来启动期压力

现状：

- `appStore.preloadIconUrls()` 会遍历全部应用并转换图标 URL。
- `getIconUrl()` 中存在调试 `console.log`。
- `iconsDir` 通过 Tauri 命令懒加载一次，后续路径由前端拼接。

建议：

- 改为按可视区域懒加载图标，应用卡片进入视口时再转换。
- 移除生产环境中的图标路径日志。
- 后端返回应用时可顺带返回图标文件名，前端统一由一个图标服务转换和缓存。
- 对图标缺失或文件被删除的情况增加兜底刷新和清理策略。

收益：

- 大量应用时首屏更快。
- 控制台噪声减少。
- 图标文件异常时更易恢复。

### 5.4 维护检测在锁内执行文件系统操作

现状：

- `validate_all_apps()` 和 `check_all_updates()` 会在持有配置锁时遍历应用并进行路径验证、元数据读取或版本检测。
- 应用数量较多、网络路径较慢或磁盘响应慢时，可能延长锁持有时间。

建议：

- 进入命令后先克隆需要检测的应用快照，释放配置锁。
- 文件系统检测在锁外执行。
- 需要写回状态时再短暂加锁，或由前端统一保存检测结果。
- 长任务支持进度事件、取消和分批处理。

收益：

- 避免长时间阻塞其他配置读写命令。
- 维护工具体验更平滑。
- 网络路径不可达时对主窗口影响更小。

### 5.5 便签和待办保存缺少防抖

现状：

- `notesStore.updateNote()` 每次内容变化都立即写入 `localStorage`。
- `todoStore` 的新增、更新、完成、删除也会完整写入所有待办。

建议：

- 便签内容编辑使用 300-800ms 防抖保存，并在窗口关闭、失焦或应用退出前强制保存。
- 待办数据量不大时可以继续完整保存，但迁移到文件后建议引入统一保存队列。
- 对保存失败给出用户可见的提示，避免静默丢失。

收益：

- 高频输入时减少写入压力。
- 数据保存失败时更容易被发现。

## 6. 使用便捷性可优化点

### 6.1 增加完整导入、导出和备份恢复

建议能力：

- 一键导出全部数据：分类、应用、图标、场景、便签、待办、剪贴板、快捷动作设置。
- 导入时支持预览差异和冲突处理。
- 支持只导入某类数据，例如只导入场景或只导入应用分类。
- 支持自动备份最近 N 份配置。

收益：

- 重装系统、迁移电脑和排查问题更方便。
- 也为未来云同步打基础。

### 6.2 优化首次使用引导

建议：

- 首次启动时提供轻量引导：
  - 拖入 `.exe`、`.lnk`、文件夹或文件。
  - 创建分类。
  - 配置快捷键。
  - 打开快捷搜索和便签。
- 在空状态区域放置直接操作入口，而不是只展示空白。
- 对维护工具的“初始化基准数据”说明更明确，说明它用于后续判断程序是否更新。

收益：

- 降低新用户理解成本。
- 让已实现能力更容易被发现。

### 6.3 提升快捷搜索能力

建议：

- 支持拼音首字母或模糊匹配，适合中文应用名。
- 支持搜索场景、便签、待办。
- 支持命令前缀：
  - `/` 网页搜索。
  - `c:` 剪贴板。
  - `n:` 便签。
  - `t:` 待办。
  - `s:` 场景。
- 支持最近启动权重、收藏权重和精确匹配权重。
- 搜索结果增加二级动作，例如管理员启动、打开所在文件夹、复制路径。

收益：

- 快捷搜索从“查找应用”升级为统一入口。
- 高频用户可以更少依赖鼠标。

### 6.4 场景工作流增加执行可控性

建议：

- 每个动作支持“失败时中断 / 失败时继续”的策略。
- 场景执行页展示每个动作的状态、耗时和错误。
- 支持手动停止正在执行的场景。
- 支持场景复制、导入导出和模板库。
- 支持动作条件，例如“程序未运行时启动”“窗口存在时发送按键”“文件存在时打开”。

收益：

- 场景失败时更容易定位问题。
- 自动化能力更接近真实工作流。

### 6.5 维护工具增强批量操作体验

建议：

- 批量删除前展示可确认的清单，包含名称、路径、失效原因。
- 更新检测支持“接受当前状态为新基准”。
- 检测结果支持导出日志。
- 网络路径检测需要显示超时原因，并允许跳过网络路径。

收益：

- 降低误删风险。
- 更新检测结果更易管理。

## 7. 可扩展点梳理

### 7.1 场景动作扩展

当前新增动作需要修改：

- `SceneActionType`
- `SCENE_ACTION_TYPES`
- `SceneAction.params`
- `scenesStore.executeAction()`
- `SceneEditor.vue` 中的配置 UI

建议：

- 定义动作注册表：
  - `type`：动作类型。
  - `schema`：参数结构。
  - `ui`：参数编辑器。
  - `executor`：执行逻辑。
  - `capability`：需要的系统权限。
- 先在内部实现注册表，不急于开放第三方插件。

适合扩展的动作：

- 打开终端并执行命令。
- HTTP 请求。
- 写入剪贴板。
- 读取文件内容。
- 条件分支。
- 循环执行。
- 调用外部脚本。

### 7.2 快捷动作扩展

当前快捷动作由 `PRESET_ACTIONS` 固定定义，执行基于 PowerShell 模板。

建议：

- 将内置动作与用户自定义动作分离。
- 用户自定义动作保存到独立配置文件。
- 动作参数使用环境变量或结构化参数，不直接拼接脚本。
- 为危险动作增加确认开关，例如结束进程、删除文件、管理员运行。

适合扩展的快捷动作：

- 打开配置文件目录。
- 复制应用名。
- 复制所在目录。
- 以指定工作目录启动。
- 启动时附带参数。
- 固定到顶部或移动到指定分类。

### 7.3 搜索源扩展

建议建立统一搜索源接口：

- `id`：搜索源标识。
- `enabled`：是否启用。
- `prefix`：可选前缀。
- `search(query)`：返回结果。
- `execute(result)`：执行结果。
- `weight`：排序权重。

可扩展搜索源：

- 应用。
- 分类。
- 场景。
- 剪贴板。
- 便签。
- 待办。
- 网页搜索。
- 本地文件。
- 系统设置。

### 7.4 数据同步与备份扩展

建议：

- 先实现本地完整导出导入。
- 再实现自动备份。
- 最后再考虑云同步。

云同步可以保留抽象接口：

- 本地文件同步。
- WebDAV。
- Git 仓库。
- 自建服务。

需要提前考虑：

- 数据冲突合并。
- 敏感数据加密。
- 图标文件同步策略。
- 剪贴板历史是否参与同步。

### 7.5 跨平台扩展

当前项目明显偏 Windows：

- 依赖 Windows API 提取图标。
- 右键菜单通过注册表实现。
- PowerShell 动作模板大量存在。
- 快捷方式解析依赖 Windows COM。
- Toast 通知和窗口等待依赖 Windows 能力。

建议：

- 把平台相关逻辑明确放入 `platform/windows`。
- 定义平台能力接口，例如：
  - 图标提取。
  - 快捷方式解析。
  - 管理员启动。
  - 进程检测。
  - 窗口查找。
  - 通知发送。
  - 右键菜单注册。
- 非 Windows 平台先返回明确错误或降级实现。

收益：

- 即使短期只支持 Windows，代码边界也更清楚。
- 后续跨平台不会被业务逻辑和平台逻辑耦合阻塞。

## 10. 总结

该项目功能基础已经比较完整，短期最值得投入的是“架构边界”和“数据可靠性”：

- 后端命令拆分能降低维护成本。
- 持久化统一能解决备份、迁移、多窗口同步和未来云同步的根问题。
- 快捷键、搜索和场景工作流的优化能直接提升日常使用效率。
- 安全边界收紧是后续开放自定义脚本、动作扩展和联网能力前必须补上的基础。

如果按上述顺序推进，项目可以从“功能丰富的本地启动器”逐步演进为“可扩展的桌面效率平台”。
