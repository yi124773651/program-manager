# 程序管理器优化改造设计书

> 本设计书参照 `docs/项目优化点梳理.md` 制定，聚焦架构治理、统一持久化、性能优化、使用便捷性和扩展能力。当前阶段不设计数据同步、云同步、WebDAV、Git 同步、自建同步服务等能力，但必须兼容已有配置数据，并为旧数据提供可验证的升级迁移路径。

## 1. 设计目标

本次优化改造的核心目标是把项目从“功能堆叠可用”推进到“边界清晰、数据可靠、便于扩展”的状态。

具体目标：

1. 拆分后端命令模块，降低 `src-tauri/src/commands/mod.rs` 的复杂度。
2. 统一本地持久化体系，将分散在 `config.json` 与多个 `localStorage` 的数据纳入可管理的数据仓储层。
3. 保持对已有 `config.json`、图标文件、旧 base64 图标和 `localStorage` 数据的兼容。
4. 提供旧数据升级流程，升级前自动备份，升级失败可回退。
5. 将全局快捷键、快捷搜索、场景动作、维护工具等能力抽象为更清晰的服务边界。
6. 收紧脚本执行、资产协议和网络访问边界。
7. 补齐关键路径测试，保证迁移、配置读写和核心工作流可回归验证。

## 2. 非目标范围

本阶段明确不做：

1. 数据同步、云同步、WebDAV 同步、Git 同步、自建服务同步。
2. 多端冲突合并。
3. 账户体系、登录、远程配置中心。
4. 第三方插件市场。
5. 完整跨平台支持。
6. 大规模 UI 重设计。

说明：

- 可以设计本地导入、导出、备份和恢复，因为它们属于本地数据可靠性能力，不属于数据同步。
- 可以预留接口边界，但不得在本阶段引入同步协议、远程账号或冲突合并逻辑。

## 3. 当前状态摘要

现有关键事实：

1. 主配置存储在 `%APPDATA%/program-manager/config.json`。
2. 图标存储在 `%APPDATA%/program-manager/icons`。
3. `Config.version` 当前默认值为 `1.0`。
4. `src-tauri/src/utils/config.rs` 已有 base64 图标迁移到文件图标的逻辑。
5. 场景、便签、待办、剪贴板历史仍由前端 Store 写入 `localStorage`：
   - 场景：`app_scenes_config`
   - 便签：`quick_notes`
   - 待办：`todo_schedule`
   - 剪贴板历史：`clipboard_history`
6. 后端命令大量集中在 `src-tauri/src/commands/mod.rs`。
7. `src-tauri/src/main.rs` 中全局快捷键仍硬编码，与设置中的快捷键字段没有形成闭环。
8. `package.json`、`Cargo.toml`、`tauri.conf.json` 存在版本号不一致。

## 4. 总体架构设计

### 4.1 分层结构

目标结构：

```text
src/
├── adapters/              # 前端外部能力适配：Tauri、剪贴板、Shell、事件
├── services/              # 前端业务服务：搜索、场景、配置、快捷键
├── stores/                # Pinia 状态，只保留响应式状态和用例调用
├── components/            # UI 组件
├── views/                 # 页面视图
└── types/                 # 类型定义

src-tauri/src/
├── commands/              # Tauri 命令入口，按业务拆分
├── services/              # 后端业务服务
├── storage/               # 本地数据仓储、迁移、备份、原子写入
├── platform/              # 平台能力，当前重点为 Windows
├── models/                # 数据模型
└── utils/                 # 通用工具
```

### 4.2 后端命令模块拆分

将 `commands/mod.rs` 拆分为：

1. `commands/config.rs`
   - `load_config`
   - `save_config`
   - `get_data_status`
   - `run_data_migration`

2. `commands/apps.rs`
   - `add_app`
   - `delete_app`
   - `launch_app`
   - `launch_app_as_admin`
   - `open_file_location`
   - `resolve_shortcut`
   - `check_app_exists`
   - `get_icons_dir`

3. `commands/scenes.rs`
   - `execute_action_template`
   - `is_process_running`
   - `send_keys`
   - `wait_for_window`

4. `commands/maintenance.rs`
   - `validate_all_apps`
   - `init_update_baseline`
   - `init_all_baselines`
   - `check_app_update`
   - `check_all_updates`
   - `batch_delete_apps`

5. `commands/integration.rs`
   - `register_context_menu`
   - `unregister_context_menu`
   - `is_context_menu_registered`
   - 后续快捷键动态注册命令

6. `commands/network.rs`
   - `fetch_image_as_base64`
   - `check_app_version_update`

兼容策略：

- 对外 Tauri 命令名称保持不变，避免前端大面积改动。
- `commands/mod.rs` 只保留模块导出和 `pub use`。

### 4.3 前端业务层拆分

新增前端服务：

1. `services/configService.ts`
   - 统一加载、保存、刷新配置。
   - 订阅后端配置变更事件。
   - 提供保存队列和防抖保存。

2. `services/searchService.ts`
   - 聚合搜索源。
   - 构建应用轻量索引。
   - 处理搜索结果排序和执行。

3. `services/sceneService.ts`
   - 执行场景动作。
   - 处理动作失败策略、进度和取消。
   - 封装系统动作调用。

4. `services/shortcutService.ts`
   - 快捷键校验。
   - 前端设置与后端注册状态同步。

5. `adapters/tauriAdapter.ts`
   - 包装 `invoke` 和事件监听。
   - 统一错误转换。

Store 调整原则：

- Store 不直接拼接 PowerShell 脚本。
- Store 不直接处理复杂业务规则。
- Store 可以保留 UI 状态，例如当前分类、搜索输入、执行进度、弹窗开关。

## 5. 本地持久化设计

### 5.1 数据目录

继续使用现有配置目录：

```text
%APPDATA%/program-manager/
├── config.json
├── scenes.json
├── notes.json
├── todos.json
├── clipboard.json
├── actions.json
├── migrations.json
├── backups/
└── icons/
```

各文件职责：

1. `config.json`
   - 应用、分类、设置、维护元数据。

2. `scenes.json`
   - 场景列表、场景动作、场景执行设置。

3. `notes.json`
   - 快捷便签。

4. `todos.json`
   - 待办日程表。

5. `clipboard.json`
   - 剪贴板历史。

6. `actions.json`
   - 用户启用的快捷动作和后续自定义动作。

7. `migrations.json`
   - 已执行迁移版本、迁移时间、迁移状态。

8. `backups/`
   - 迁移前自动备份。
   - 手动导出和自动备份可复用该目录结构。

### 5.2 数据文件格式

所有数据文件统一采用以下外壳结构：

```json
{
  "schemaVersion": 1,
  "updatedAt": 1780000000000,
  "data": {}
}
```

设计原因：

- `schemaVersion` 用于后续迁移。
- `updatedAt` 用于本地诊断和备份排序。
- `data` 承载业务数据，避免未来新增元信息影响业务结构。

### 5.3 原子写入

所有后端文件写入使用以下流程：

1. 序列化目标 JSON。
2. 写入同目录临时文件，例如 `config.json.tmp`。
3. 刷新文件内容到磁盘。
4. 用临时文件替换正式文件。
5. 替换成功后删除临时文件残留。

失败策略：

- 正式文件不被破坏。
- 下次启动检测 `.tmp` 文件并清理。
- 迁移期间失败时优先使用备份恢复。

### 5.4 配置兼容设计

需要兼容的数据：

1. 现有 `config.json`。
2. 旧 base64 图标。
3. 已迁移到文件名的图标。
4. 缺少新增字段的旧 `AppSettings`。
5. 缺少 `itemType` 的旧应用条目。
6. `localStorage` 中的场景、便签、待办、剪贴板数据。

兼容原则：

- 读取旧数据时不报错。
- 缺失字段使用默认值补齐。
- 无法迁移的局部数据记录错误，但不阻止主应用启动。
- 迁移前必须备份。
- 迁移成功后写入 `migrations.json`，避免重复迁移。

## 6. 旧数据升级设计

### 6.1 迁移版本

定义迁移版本：

```text
1.0.0 -> 1.1.0：补齐 config 字段，保持现有 config.json 结构。
1.1.0 -> 1.2.0：将 scenes、notes、todos、clipboard 从 localStorage 迁移到独立 JSON 文件。
1.2.0 -> 1.3.0：引入 actions.json，迁移用户启用动作配置。
```

实际版本号可结合发布版本调整，但迁移编号必须单调递增。

### 6.2 启动迁移流程

应用启动时：

1. 后端加载 `config.json`。
2. 执行后端可完成的迁移：
   - 字段补齐。
   - 图标迁移。
   - schemaVersion 初始化。
3. 前端启动后读取 `localStorage` 旧数据状态。
4. 前端调用后端迁移命令，把旧 `localStorage` 数据写入对应 JSON 文件。
5. 后端写入 `migrations.json`。
6. 前端保留旧 `localStorage` 数据一段兼容期，只标记为已迁移，不立即删除。

### 6.3 为什么 localStorage 迁移需要前端参与

Tauri 后端不能直接可靠读取 WebView 的 `localStorage`，因此迁移流程应由前端读取旧数据，再通过 Tauri 命令提交给后端落盘。

前端提供迁移负载：

```ts
interface LegacyLocalStoragePayload {
  scenes?: unknown
  notes?: unknown
  todos?: unknown
  clipboard?: unknown
}
```

后端负责：

- 校验结构。
- 补齐默认值。
- 写入目标 JSON。
- 记录迁移状态。

### 6.4 迁移备份

迁移前生成备份目录：

```text
backups/migration-YYYYMMDD-HHmmss/
├── config.json
├── scenes.localStorage.json
├── notes.localStorage.json
├── todos.localStorage.json
├── clipboard.localStorage.json
└── manifest.json
```

`manifest.json` 记录：

- 应用版本。
- 迁移开始时间。
- 迁移目标版本。
- 已备份文件列表。
- localStorage 键名和数据长度。

### 6.5 迁移失败处理

失败分类：

1. `config.json` 解析失败：
   - 复制原文件到备份目录。
   - 使用默认配置启动。
   - UI 显示配置损坏提示。

2. 单个 localStorage 数据解析失败：
   - 跳过该数据。
   - 记录迁移错误。
   - 其他数据继续迁移。

3. 目标 JSON 写入失败：
   - 保留旧 `localStorage` 数据。
   - 标记迁移未完成。
   - 下次启动重试。

4. 图标迁移失败：
   - 保留原 icon 字段。
   - 应用可继续启动。
   - 记录失败项，后续维护工具可清理。

## 7. 快捷键设计

### 7.1 配置来源

快捷键配置继续保存在 `config.json` 的 `settings` 中：

- `globalShortcut`
- `spotlightShortcut`
- `quickNotesShortcut`
- 后续新增 `todoScheduleShortcut`

### 7.2 注册流程

1. 启动时读取配置。
2. 后端注册快捷键。
3. 注册结果写入运行态状态，不写入配置。
4. 设置变更时：
   - 前端保存配置。
   - 调用后端重新注册。
   - 后端注销旧快捷键，再注册新快捷键。
5. 注册失败时恢复旧快捷键，并返回错误。

### 7.3 UI 行为

设置界面提供：

- 快捷键录制输入。
- 冲突提示。
- 恢复默认按钮。
- 开关状态与快捷键注册状态区分展示。

## 8. 搜索设计

### 8.1 搜索源接口

```ts
interface SearchProvider {
  id: string
  name: string
  enabled: () => boolean
  prefix?: string
  search: (query: string) => Promise<SearchResult[]>
  execute: (result: SearchResult) => Promise<void>
  weight: number
}
```

首批内置搜索源：

1. 应用。
2. 剪贴板。
3. 网页搜索。
4. 计算器。
5. 场景。
6. 便签。
7. 待办。

### 8.2 配置同步

不再每次搜索都调用 `load_config`。

改为：

1. 主窗口配置保存成功后发出 `config-changed` 事件。
2. 搜索窗口收到事件后刷新配置快照。
3. 搜索窗口打开时主动加载一次配置。
4. 搜索过程中只使用本地快照和索引。

### 8.3 索引策略

应用索引字段：

- `id`
- `name`
- `nameLower`
- `path`
- `pathLower`
- `lastLaunched`
- `iconUrl`

后续可增加：

- 拼音首字母。
- 别名。
- 标签。

## 9. 场景工作流设计

### 9.1 动作注册表

```ts
interface SceneActionDefinition {
  type: string
  name: string
  description: string
  paramsSchema: unknown
  defaultParams: Record<string, unknown>
  capability: string[]
  execute: (context: SceneExecutionContext) => Promise<SceneActionResult>
}
```

首期仍保留现有动作类型：

- 启动程序。
- 管理员启动。
- 打开网页。
- 打开文件夹。
- 打开文件。
- 关闭程序。
- 延迟等待。
- 发送通知。

### 9.2 执行控制

新增执行状态：

- `pending`
- `running`
- `success`
- `failed`
- `skipped`
- `cancelled`

新增动作失败策略：

- `continue`：失败后继续执行后续动作。
- `stop`：失败后中断场景。

新增场景能力：

- 复制场景。
- 导入导出单个场景。
- 查看执行日志。
- 手动取消执行。

## 10. 安全设计

### 10.1 PowerShell 执行边界

设计原则：

1. 业务层不直接拼接未转义用户输入。
2. 参数优先通过环境变量传入。
3. 执行器统一记录命令类型、耗时、退出码、标准输出和错误输出。
4. 高风险动作需要明确 UI 确认。

高风险动作：

- 管理员运行。
- 结束进程。
- 删除文件。
- 执行用户自定义脚本。

### 10.2 资产协议范围

将 `assetProtocol.scope` 从宽泛的 `["**"]` 收窄到：

- 图标目录。
- 背景图片目录。
- 必要的应用资源目录。

### 10.3 网络访问

本阶段保留：

- 背景图片获取。
- GitHub Releases 更新检查。

要求：

- 必须有超时。
- 必须校验响应类型。
- 错误必须可见。
- 不引入数据同步相关网络能力。

## 11. 导入导出与备份设计

本阶段支持本地导入导出，不支持同步。

导出包结构：

```text
program-manager-backup-YYYYMMDD-HHmmss.zip
├── manifest.json
├── config.json
├── scenes.json
├── notes.json
├── todos.json
├── clipboard.json
├── actions.json
└── icons/
```

导入流程：

1. 读取 `manifest.json`。
2. 校验版本和数据结构。
3. 展示导入预览。
4. 自动备份当前数据。
5. 执行覆盖导入或选择性导入。
6. 刷新前端状态。

## 13. 验收标准

设计落地后应满足：

1. 旧 `config.json` 可以正常加载。
2. 旧 base64 图标可迁移为文件，失败时不影响应用启动。
3. 旧 `localStorage` 场景、便签、待办和剪贴板数据可迁移到本地 JSON 文件。
4. 迁移前自动备份，迁移失败不破坏旧数据。
5. 后端命令按领域拆分，命令名称对前端保持兼容。
6. 搜索窗口不再每次输入都刷新完整配置。
7. 快捷键配置可以动态注册和恢复默认。
8. 本地导入导出可覆盖主要数据文件和图标。
9. 数据同步相关能力未进入本阶段实现范围。
10. 自动化测试覆盖迁移、配置读写、搜索和场景核心路径。
